> **내부 엔티티 식별자와 외부에 노출되는 식별자를 같은 것으로 두어도 될까?**

프로젝트의 식별자 설계에서 가장 먼저 고민한 내용이었습니다.  
결론은, 해당 프로젝트에서는 **같이 두지 않는다**는 쪽으로 정하고 진행했습니다.

도메인 모델 관점에서 엔티티는 생명주기 동안 상태가 크게 바뀌더라도 동일한 존재로 추적될 수 있어야 합니다.<br>  
그래서 시스템 내부에는 변하지 않는 **식별자**가 당연히 필요하고 프로젝트에서 일반적인 **Long 타입의 PK**를 사용했습니다.

PostgreSQL에서는 `generated by default as identity`를 사용했고, JPA에서는 아래처럼 `IDENTITY` 전략으로 매핑했습니다.

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)

대신 외부 요청과 응답, URL path, 파라미터에는 **별도의 공개 식별자**를 노출하는 구조를 적용했습니다.

예를 들면 사용자와 채팅 세션 같은 도메인에서 다음처럼 역할을 나눴습니다.

| 구분 | 용도 | 예시 |
| --- | --- | --- |
| 내부 식별자 | DB 조인, 연관관계, 영속성 관리 | `id` |
| 외부 식별자 | API 요청/응답, URL path, 클라이언트 참조 | `userToken`, `sessionToken` |

<!--more-->

---

## 왜 굳이 분리했나

처음에는 이 설계를 단순히 **"id를 숨기기 위해서"** 적용했습니다.  
하지만 프로젝트를 진행하면서 이유를 다시 정리해보니 식별자 은닉만큼 중요한 다른 점도 많았습니다.

> **내부에서 쓰는 식별자와 외부에 보여주는 식별자의 역할을 분리하고 싶었다.**

순차적 증가 방식의 내부적인 PK가 외부 API에 그대로 노출되면 다음과 같은 문제가 생길 수 있습니다.

- 리소스 개수나 생성 순서를 유추하기 쉽습니다.
- 예측 가능한 식별자를 바탕으로 다른 객체 조회를 시도해보는 공격이 더 쉬워질 수 있습니다.
- 외부에서 사용하는 값이 내부 DB 구조와 너무 강하게 연결됩니다.

물론 숫자 id가 노출된다고 해서 곧바로 취약점이 생기는 것은 아니겠지만 공개 식별자가 순차적인 숫자일수록 다른 리소스를 추측해 조회해보는 시도가 쉬워지는 것은 분명하다고 생각했습니다.

그래서 외부에 공개하는 식별자는 내부 PK와 역할을 분리하는 편이 더 안전하고 나중에 내부 구현이 바뀌더라도 외부에서 사용하는 방식은 비교적 안정적으로 유지하기 쉽다고 생각했습니다.

---

## 구현은 어떻게 했나

구현에서는 공개 식별자를 **문자열 token** 으로 생성하고, `unique` 제약을 두는 방식으로 구현했습니다.

- 공통 유틸에서 `SecureRandom` 기반 영숫자 문자열 생성
- 성격이 다른 값은 다른 prefix 부여
- 외부 요청/응답과 URL에는 token 사용
- 내부 영속성 및 관계 관리는 숫자 PK 유지

예를 들면 이런 느낌입니다.

    userToken    = usr_a8K3mP...
    sessionToken = ses_x92LmQ...

이 방식의 장점은 두 가지였습니다.

1. 외부 식별자를 읽었을 때 다른 엔티티의 식별자를 유추하기 어렵게 만들 수 있습니다.
2. prefix가 있어서 UUID보다 사람이 읽거나 디버깅할 때 더 단순합니다.

또한 JWT subject에도 내부 PK가 아니라 `userToken`을 넣어 인증 이후에도 애플리케이션이 같은 기준의 식별자를 계속 사용할 수 있도록 했습니다.

---

## 성능은 괜찮았나

여기서 한 번 다시 생각할 필요가 있었습니다.

> **문자열 기반 대체키는 숫자 PK보다 비효율적이지 않나?**

실제로 랜덤성이 큰 문자열은 숫자 PK보다 다음 측면에서 불리할 수 있습니다.

- 인덱스 크기
- 비교 비용
- 인덱스 locality
- 삽입 패턴이 흩어질 때 생길 수 있는 저장 효율 저하

이 지점을 검토하면서 [Percona의 글](https://www.percona.com/blog/store-uuid-optimized-way/)도 참고했습니다.  
이 글은 `MySQL InnoDB` 환경에서 `UUID` 같은 랜덤 키를 다룰 때 어떤 비용이 생길 수 있는지를 설명합니다.

글에서는 25,000건씩 반복 삽입해 총 2,500만 건을 넣으며 UUID PK, BIGINT PK, reordered UUID PK를 비교합니다.  
이 실험에서 전달하는 핵심은 다음과 같았습니다.

- 랜덤성이 큰 값은 순차적인 숫자 PK보다 인덱스 효율이 떨어질 수 있습니다.
- 특히 문자열 기반 UUID를 PK로 사용하면 인덱스 크기와 쓰기 비용 측면에서 더 불리해질 수 있습니다.
- reordered UUID처럼 삽입 순서를 조금 더 정렬 친화적으로 만드는 방식은 이런 비효율을 완화하는 데 도움이 될 수 있습니다.

다만 여기서 말하는 `reordered UUID`와 제가 사용한 `prefix 기반 token`은 의도가 다릅니다.  
Percona 글에서 reordered UUID는 **삽입 성능과 인덱스 locality를 개선하기 위한 방식**에 가깝고, 제가 붙인 prefix는 **식별자 종류를 구분하고 가독성을 높이기 위한 방식**에 가깝습니다.

즉, `usr_`, `ses_` 같은 prefix를 붙였다고 해서 reordered UUID처럼 저장 효율이 좋아진다고 보기는 어렵습니다.  
그래서 이 프로젝트에서는 Percona 글의 내용을 "prefix를 붙였으니 랜덤 문자열도 충분히 괜찮다"는 근거로 사용한 것은 아니고 **무작위 문자열을 내부 PK로까지 쓰는 것은 신중해야 한다**는 쪽으로 받아들였습니다.

다만 이 프로젝트는 **PostgreSQL** 을 사용했고, Percona 글은 **MySQL InnoDB의 클러스터드 PK 특성**을 전제로 합니다.  
그래서 그 내용을 그대로 적용한 것은 아니고, 다음 정도의 원칙만 참고했습니다.

- 랜덤 문자열은 숫자 PK보다 일반적으로 인덱스 측면에서 불리할 수 있습니다.
- 그렇다면 내부 PK는 숫자로 유지하는 편이 낫습니다.
- 외부 공개용 대체키는 `PK`가 아니라 `unique column`으로 두는 쪽이 균형이 좋습니다.

즉, 랜덤 문자열의 비용을 모르고 도입한 것이 아니라, **비용을 알고도 감수 가능한 위치에만 배치한 것**에 가깝습니다.

> **대체키를 쓰는 것 자체보다, 그 대체키를 어디에 두느냐가 더 중요하다고 생각했습니다.**

---

## 장점

### 1. 외부에 보여주는 값과 내부 구현을 분리할 수 있습니다

나중에 저장소 구조나 내부 엔티티 설계가 바뀌더라도, 외부에서 사용하는 식별 방식은 상대적으로 덜 흔들릴 수 있습니다.

### 2. 프론트엔드와의 기준이 단순해집니다

path, param, response에서 일관되게 공개 식별자를 사용하면, 어떤 값을 참조 기준으로 삼아야 하는지가 분명해집니다.

### 3. 식별자의 역할을 구분하는 습관이 생깁니다

내부 숫자 PK를 외부에 그대로 노출하지 않는다는 원칙이 생기면서, 어떤 값이 **시스템 내부 식별자**이고 어떤 값이 **외부 참조용 값**인지 더 분명하게 구분하게 됐습니다.

특히 채팅 세션처럼 외부에서 직접 참조될 가능성이 높은 도메인에서는 이런 분리가 더 의미 있었다고 느꼈습니다.

---

## 아쉬운 점

설계 의도는 분명히 이랬습니다.

> **외부는 `token`, 내부는 `id`**

그런데 구현을 진행하면서 조회 로직 상당수가 token 중심으로 흘러갔습니다.  
그 결과 서버 내부에서도 숫자 PK를 충분히 활용하지 못했고, 어느 순간에는 이런 생각도 들었습니다.

> 이럴 거면 그냥 token을 PK처럼 써도 되는 것 아닌가?

지금 돌아보면 문제는 **대체키를 도입한 것 자체**보다, **대체키의 사용 범위를 명확히 통제하지 못했던 것**에 더 가까웠습니다.

원래 더 깔끔하게 가져갔어야 했던 방향은 아마 이랬을 것입니다.

1. 컨트롤러나 인증 경계에서는 `token`으로 요청을 받습니다.
2. 애플리케이션 초입에서 이를 내부 `id`로 변환합니다.
3. 이후 도메인 로직, 조인, 연관관계는 `id` 중심으로 처리합니다.

이렇게 했더라면 외부 식별자 분리의 장점을 유지하면서도, 내부 구현 복잡도는 더 잘 관리할 수 있었을 것 같습니다.

특히 [Percona의 글](https://www.percona.com/blog/store-uuid-optimized-way/)을 다시 참고하고 나서는 이 아쉬움이 더 분명해졌습니다.  
**랜덤 문자열을 외부 공개 식별자로 쓰는 것**과 **내부 PK 역할까지 맡기는 것**은 전혀 다른 문제였기 때문입니다.

---

## 지금 다시 만든다면

지금 다시 같은 구조를 만든다면 **대체키 자체는 유지할 것 같습니다.**  
다만 다음에는 역할을 훨씬 더 분명하게 나눌 것 같습니다.

| 영역 | 사용할 식별자 |
| --- | --- |
| API 경계, URL, 응답 | 공개 식별자 (`token`) |
| 내부 도메인 흐름, 조인, 연관관계 | 내부 식별자 (`id`) |

결국 중요한 건 대체키를 쓰느냐 마느냐보다, **대체키를 어디까지 사용할 것인가를 명확히 정하는 것**이었습니다.

외부 공개 식별자는 외부에서 리소스를 식별하기 위해 존재하고, 내부 PK는 내부 구현을 안정적으로 유지하기 위해 존재합니다.  
이 둘의 역할이 섞이기 시작하면 설계의 장점도 흐려집니다.

이번 경험을 통해 배운 것도 바로 그 점이었습니다.
