## 자바 코드로 CAS 구현해보기
실제 자바의 AtomicInteger 내부는 네이티브 코드(C/C++)로 구현되어 있다. 

이해하기 쉽게 자바 코드로 단순하게 구현해보자.

```java
public class MyAtomicInteger {
    private volatile int value; // 메모리 가시성 보장

    // CPU의 CAS 명령어를 호출한다고 가정하는 메서드 (실제로는 Unsafe 클래스 사용)
    // 메모리 값이 expectedValue와 같으면 newValue로 바꾸고 true 반환, 다르면 false 반환
    private boolean compareAndSet(int expectedValue, int newValue) { ... }

    // 원자적 증가 메서드
    public int incrementAndGet() {
        int expectedValue;
        int newValue;
        
        do {
            expectedValue = this.value;     // 1. 메모리에서 현재 값 읽기
            newValue = expectedValue + 1;   // 2. 내가 덮어쓸 새로운 값 계산
            
        // 3. CAS 실행
        // 만약 성공(true)하면 !true -> false가 되어 루프 탈출.
        // 만약 실패(false)하면 !false -> true가 되어 다시 do 처음으로 돌아감(재시도).
        } while (!compareAndSet(expectedValue, newValue)); 
        
        return newValue;
    }
}
```

1. `compareAndSet()`: 자바에서 CAS를 실행하는 메서드 이름 비교하고(Compare) 세팅(Set)을 수행

2. `do-while` 루프의 : * 스레드가 락(Lock)을 기다리며 Blocked 되지 않음. (Non-blocking)
<br> compareAndSet이 true를 반환할 때까지 계속 루프를 돕니다. 중간에 다른 스레드가 값을 가로채면 실패(false)가 떨어지고, 루프를 다시 돌면서 최신 값을 다시 읽어옴.

## CAS의 치명적인 단점
### 장점
충돌이 적은 일반적인 상황에서는 스레드를 Block하고 깨우는 비용(Context Switching)이 없기 때문에 synchronized보다 빠름.

### 단점 (주의점)
만약 **수백, 수천 개의 스레드가 동시에 단 하나의 변수만 빈번하게 수정하려는 극단적인 상황(High Contention)** 에는 문제 발생

하나의 스레드만 성공하고 나머지 수백 개는 실패해서 루프를 반복.

Block되지 않고 계속 CPU를 쓰면서 루프를 돌기 때문에 의미 없는 재시도로 CPU 코어 사용률이 100%를 찍으면 프로세스가 종료될 수 있음

이런 상황에서는 synchronized로 락을 걸고 순서대로 처리하는 게 전체 성능에 더 좋을 수 있음
