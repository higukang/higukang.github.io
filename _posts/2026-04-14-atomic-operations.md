## 원자적 연산(Atomic Operation)
 <!--more-->
원자적(Atomic): 

- **물리학에서의 원자(Atom)**: '더 이상 쪼개질 수 없는 가장 작은 단위' 
- **프로그래밍에서 원자적 연산(Atomic Operation)**: 중단되지 않고 한 번에(통째로) 실행되는 연산


## ++의 연산 과정
자바의 `++` 연산은 한 줄이지만 3단계를 거쳐 연산이 수행됨
```java
int count;
count++;
count = count + 1;
```
1. Read (읽기): 메인 메모리에서 count의 현재 값을 CPU로 읽어온다.
2. Modify (수정): CPU 안에서 그 값에 1을 더한다. (0 + 1 = 1)
3. Write (쓰기): 계산된 결과(1)를 다시 메인 메모리의 count에 저장한다.

## 멀티스레드 환경 레이스 컨디션 (Race Condition)
### 10개의 스레드가 동시에 이 count++를 실행하면 되면?
> 스레드들은 CPU를 번갈아 가며 사용하기 때문에(컨텍스트 스위칭), 다음과 같은 끔찍한 타이밍 스텝이 꼬일 수 있다.

1. 스레드 A가 메모리에서 count 값 0을 읽어옴. (1단계)
2. 스레드 A가 1을 더하려고 하는 찰나에, CPU 제어권이 스레드 B로 넘어간다.
3. 스레드 B도 메모리에서 count 값을 읽어옵니다. (아직 A가 쓰기를 안 했으니 여전히 count는 0) -> 여기서부터 문제 발생
4. 스레드 B가 0에 1을 더해 메모리에 1을 쓴다. (count = 1)
5. 다시 스레드 A가 아까 멈췄던 작업을 이어서 한다. 자기가 읽어뒀던 0에 1을 더해 메모리에 1을 쓴다. (count = 1)

분명히 두 개의 스레드가 count++를 호출했으니 결과가 2가 되어야 하는데, 중간에 쪼개진 틈으로 다른 스레드가 파고들면서 결과가 1이 됨.
<br>백엔드 서버에서 결제 금액이나 재고 수량을 이렇게 처리한다면 치명적인 버그가 발생하게 됨

## 정리
- 원자적 연산은 쪼개질 수 없는 단일 연산이다.

- 우리가 자주 쓰는 i++는 3단계(Read-Modify-Write)로 이루어져 있어 원자적이지 않다.

- 멀티스레드 환경에서 그냥 i++를 쓰면 데이터 유실(경쟁 상태, Race Condition)이 발생한다.
