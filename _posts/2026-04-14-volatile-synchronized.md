> count++ 같은 연산이 내부적으로는 3단계(Read-Modify-Write)로 이루어져 있어 원자적이지 않기때문에 <br>여러 스레드가 동시에 접근할 때 '경쟁 상태(Race Condition)'가 발생해 데이터가 유실될 수 있다.

## 메모리 가시성과 원자성
동시성 문제를 해결하려면 **메모리 가시성**과 **원자성**을 알아야한다.

<!--more-->

### 1. 메모리 가시성 (Memory Visibility) 문제와 volatile
스레드들은 성능을 높이기 위해 메인 메모리(RAM)에서 변수 값을 읽어온 후 자신만의 **CPU 캐시(Cache)** 에 저장해 두고 사용한다

문제는 스레드 A가 값을 변경해서 캐시에만 저장하고 메인 메모리에 반영하지 않았다면 스레드 B는 여전히 메인 메모리의 옛날 값을 읽어오게 된다. 
<br> 이를 `메모리 가시성 문제`라고 한다.

`volatile` 키워드의 역할: 
<br>변수 앞에 volatile을 붙이면, 스레드들이 CPU 캐시를 무시하고 무조건 메인 메모리에서만 값을 읽고 쓰도록 강제한다.

```java
public volatile int count = 0;
```

이랗게 하면 메모리 가시성 문제는 해결된다. 
스레드 A가 값을 변경하면 즉시 메인 메모리에 반영되므로 스레드 B도 변경된 최신 값을 볼 수 있다.

> **하지만 원자성 문제는 해결하지 못한다**

- volatile은 "최신 값을 보여준다"까지만 보장한다. 
- 여러 스레드가 동시에 count++ (Read-Modify-Write 3단계)를 실행할 때, 그 3단계가 '한 번에' 실행되도록 막아주지는 못한다. 
- volatile만으로는 경쟁 상태를 완벽히 해결할 수 없다.

### 2. 원자성 문제와 synchronized
이 원자성 문제까지 해결하는 가장 강력하고 고전적인 방법은 `synchronized` 키워드를 붙이는 것이다.

```java
public synchronized void increment() {
    count++;
}
```

`synchronized` 키워드의 역할:
메서드에 이 키워드를 붙이면 해당 영역은 **'임계 영역(Critical Section)'**이 되어 한 번에 오직 하나의 스레드만 이 영역에 들어갈 수 있도록 **'락(Lock)'**을 건다.

`synchronized`는 단순히 락만 거는 게 아니라, 내부적으로 다음 두 가지를 모두 보장한다:

> 1. **메모리 가시성** (`Visibility`)<br>
	•	`synchronized` 블록에 들어갈 때 → 메인 메모리 값 읽음<br>
    •	나올 때 → 변경 내용을 메인 메모리에 `flush`
> 2.  그래서 다른 스레드도 항상 최신 값을 보게 됨, `volatile`의 역할을 이미 포함하고 있음

- **장점**: 스레드 안전성(`Thread-Safety`)을 보장한다. 메모리 가시성 문제와 원자성 문제를 한 번에 해결한다.

- **단점 (성능 저하)**: 한 스레드가 `lock`을 얻고 `increment()`를 실행하는 동안 다른 스레드들은 대기(`BLOCKED` 상태)해야 한다. 
<br>트래픽이 몰리는 스프링 부트 서버 환경이라면, 이런 병목 현상이 발생하면 애플리케이션 전체의 성능이 저하된다.

## 정리
| 구분 | 메모리 가시성 (Visibility) | 원자성 (Atomicity) | 특징 |
|------|---------------------------|-------------------|------|
| 일반 변수 (`int count`) | X | X | 데이터 유실 발생 (Race Condition) |
| `volatile` 변수 | O | X | 최신 값은 보장되지만, `count++` 같은 연산에서는 여전히 데이터 유실 |
| `synchronized` 사용 | O | O | 스레드 안전성 보장하지만, 락으로 인해 성능 저하 가능 |

> "성능을 떨어뜨리지 않으면서 데이터도 안전하게 보호할 수 없을까?"라는 고민이 필요함

**`락(Lock)`을 걸지 않고(`Non-blocking`) 원자성을 보장하는 방법**이 필요
