프로젝트 중 어떤 데이터를 MySQL데이터베이스에 저장하려 했습니다.

해당 데이터들은 "SV4K"와 같은 이미 존재하는 코드로 이미 식별을 할 수 있는 데이터입니다.

여기서 고민이 되는 점은  MySQL의 PK의 속성을 정할 때 "SV4K"같이 기존의 코드 값으로 PK를 정할 지<br>
아니면 내부적으로 MySQL의 BIGINT, AUTO INCREMENT로 정할 지를 정해야했습니다.

개발 편의성이나, 직접 저장해야하는 과정에서 기존에 있는 코드를 쓰는 것이 저장 하거나 개발 과정에서는 
더 편하다고 생각을 했는데<br> 
`BIGINT AUTO_INCREMENT`를 PK로 잡는 것이 성능과 저장 공간 효율상 가장 좋다고 합니다.

## InnoDB의 클러스터드 인덱스(Clustered Index)

> MySQL의 내부 엔진인 InnoDB는 PK가 데이터 저장 순서입니다

예를 들어 PK가 AUTO_INCREMENT라면
``` sql
id
--
1
2
3
4
5
```

실제 디스크의 B+Tree 리프 노드도 거의 이런 순서로 저장됩니다.
``` text
[1][2][3][4][5]
```

그래서 새 데이터가 들어오면 
```sql
INSERT INTO table1 VALUES (6, ...)
```
이렇게 바로 맨 뒤에 붙으면 됩니다.
``` text
[1][2][3][4][5][6]
```

그래서 아래와 같은 장점이 있습니다.
* 기존 데이터 이동 없음
* 페이지 분할(Page Split) 거의 없음
* 디스크 쓰기 최소화

## 페이지란?
InnoDB는 데이터를 한 줄(Row)씩 저장하지 않습니다.

실제로는 16KB 크기의 페이지(Page) 단위로 저장합니다.

```
페이지 1

[1]
[2]
[3]
[4]
```
이렇게 여러 레코드가 한 페이지에 들어있습니다.

### 페이지가 꽉찬다면? 
현재 페이지가 만약 아래데이터로 가득 차있고
여기에 `B`를 넣어야하는 상황이라면 어떻게 될까요

```
페이지 1

[A]
[K]
[P]
```
정렬 순서를 유지해야하므로 아래처럼 되어야하지만, 

```
[A]
[B]
[K]
[P]
```

공간이 이미 꽉차있었기 떄문에 
```
페이지1          페이지2

[A]             [K]
[B]             [P]
```
처럼 페이지를 둘로 나누게 됩니다.

이것을 `페이지 분할`이라고 합니다.

이 과정에서

1. 새 페이지 생성
2. 기존 데이터 일부 복사
3. 부모 B+Tree 노드 수정
4. 디스크 쓰기

가 발생하게 됩니다. 그래서 `INSERT` 쿼리 한 번에

`INSERT` + 구조 변경 이 발생합니다.



## UUID나 “SV4K” 같은 랜덤 PK

예를 들어 이런 같은 PK를 가지고 있는 테이블에서
```
[A123]
[K928]
[P531]
```
아래와 같은 
```
[B222]
```
데이터가 들어온다면 정렬 순서를 유지해야 하므로
```
[A123]
[B222]
[K928]
[P531]
```
이 되어야 합니다.

그런데 B+Tree의 해당 페이지가 꽉 차 있으면 
`페이지 분할(Page Split)`이 발생합니다 

```
기존

[A][K][P]

↓

분할

[A][B]
[K][P]
```

이 과정에서

* 페이지 생성
* 데이터 복사
* 부모 노드 수정

이 일어나고 랜덤 값이면 이런 일이 계속 발생합니다.

## MySQL의 AUTO_INCREMENT

AUTO_INCREMENT는 항상 맨 오른쪽 리프에만 삽입합니다

내부 B+Tree를 간단히 보면 아마 이런식으로 추가가 될 것입니다.

```
      [100]
     /     \
 1~99     100~
```
현재 최대 PK가 150이라고 하면 `id=151`은 무조건 맨 오른쪽 페이지에 들어갈 것입니다.

그래서 InnoDB는 `Right-most insert`를 효율적으로 처리할 수 있습니다.

## Secondary Index

그리고 InnoDB는 Secondary Index에 

```
Secondary Key + PK
```
를 저장합니다.

``` sql
id BIGINT PK
name VARCHAR(20)

INDEX idx_name(name)
```

실제 저장 형태
```
(name, id)
```

그래서 PK가 BIGINT라면 내부적으로 

name에 8byte만 저장하면 됩니다. 

그런데 만약 PK가 문자열이라면 
```sql 
id VARCHAR(20)
```
모든 Secondary Index가 커지게 됩니다.

## InnoDB의 캐시 효율
InnoDB Buffer Pool은 페이지 단위로 읽습니다.

PK가 BIGINT면: 8byte
PK가 UUID라면: 16byte
PK가 VARCHAR(20)이면: 20byte 이상

그래서 만약 같은 16kb의 페이지가 있다면 BIGINT로 가장 많은 레코드를 저장할 수 있습니다.

그래서 PK의 사이즈가 커질 수록, 
* 메모리 사용량 증가
* 캐시 적중률 감소
* 인덱스 깊이 증가
같은 단점이 발생하게 됩니다.

## 선택한 방식
```sql
id BIGINT AUTO_INCREMENT PRIMARY KEY
```
를 내부 PK로 사용하고

```sql
code VARCHAR(10) UNIQUE
```
을 별도로 두는 것으로 결정했습니다.

```sql
CREATE TABLE pack (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    code VARCHAR(10) NOT NULL UNIQUE
);
```

그래서 jpa 코드에서는 
```java
repository.findByCode("SV4K")
```

로 조회하고, DB 내부적으로는 `PK = BIGINT`를 사용하게 됩니다.

이런 방식으로 
* 개발자는 의미 있는 코드 사용 가능
* DB는 가장 효율적인 PK 사용 가능

이라는 두가지 장점을 동시에 가질 수 있습니다.
